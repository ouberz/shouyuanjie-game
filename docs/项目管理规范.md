## 📊 GitHub Projects看板设置

### 创建看板

1. 访问仓库Projects页面：
   `https://github.com/ouberz/shouyuanjie-game/projects`

2. 点击 **New Project**，选择 **Board** 模板

3. 创建以下列：

| 列名 | 说明 | 使用规则 |
|------|------|----------|
| **Backlog** | 待规划任务池 | 新Issue默认进入此列 |
| **To Do** | 本迭代待开发 | 确认要做的任务 |
| **In Progress** | 正在开发 | 开始工作时移动到此列 |
| **Review** | 待审核/测试 | 完成后等待Review |
| **Done** | 已完成 | 关闭Issue后自动进入 |

### 看板使用流程

```
新Issue创建
    ↓
Backlog（待评估）
    ↓
To Do（确认要做）
    ↓
In Progress（开始开发）
    ↓
Review（提交审核）
    ↓
Done（关闭Issue）
```

## 🔄 开发工作流

### 1. 创建任务Issue

**模板**：
```markdown
# 任务：[任务名称]

## 🎯 任务描述
[简要描述任务目标]

## 📝 详细步骤
- [ ] 步骤1
- [ ] 步骤2
- [ ] 步骤3

## ✅ 验收标准
- [ ] 标准1
- [ ] 标准2

## 📦 交付物
- 文件1
- 文件2

## ⏱️ 预估时间
X天

## 🏷️ 标签
Milestone, 优先级, 类型
```

### 2. 开发流程

1. **认领任务**：在Issue中评论 `/assign @your-username`
2. **创建分支**：`git checkout -b feature/issue-编号-简短描述`
3. **开发**：按照Issue中的步骤实施
4. **提交代码**：提交信息格式：`feat: 实现XXX功能 (#Issue编号)`
5. **创建PR**：关联Issue，填写PR模板
6. **Code Review**：等待审核通过
7. **合并代码**：Squash merge到develop分支
8. **关闭Issue**：合并后自动关闭

### 3. 提交信息规范

遵循 **Conventional Commits** 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type类型**：
- `feat`: 新功能
- `fix`: 修复Bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具链

**示例**：
```
feat(combat): 实现格挡弹反机制 (#5)

- 添加完美格挡判定逻辑
- 实现弹反反击系统
- 添加战斗特效

Closes #5
```

## 🌿 分支管理策略

### 分支结构

```
main（生产环境）
  ↑
develop（开发主分支）
  ↑
feature/xxx（功能分支）
hotfix/xxx（紧急修复）
release/x.x（发布分支）
```

### 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能分支 | `feature/issue-编号-描述` | `feature/issue-5-combat-system` |
| 修复分支 | `fix/issue-编号-描述` | `fix/issue-10-lifespan-bug` |
| 发布分支 | `release/版本号` | `release/0.1.0` |
| 热修复分支 | `hotfix/版本号-描述` | `hotfix/0.1.1-critical-fix` |

## 🤝 协作规范

### Code Review规范

1. **必须Review的内容**：
   - 功能逻辑正确性
   - 代码风格一致性
   - 蓝图节点命名规范
   - 注释完整性

2. **Review流程**：
   - 自查代码 → 提交PR → 等待Review → 修改反馈 → 合并代码

3. **Review Checklist**：
   - [ ] 功能符合Issue要求
   - [ ] 无明显性能问题
   - [ ] 蓝图命名规范
   - [ ] 必要注释完整
   - [ ] 无编译警告
